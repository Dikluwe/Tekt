# /00_nucleo — The Seed

> The system's causal record. Every AI-generated code must be traceable to a prompt here.

---

## Purpose

This directory contains the **structured prompts** that originated each component of the system, and the **ADRs** that document decisions affecting multiple components or the global structure.

The prompt does not describe the code — it is the origin of the code. The interface a component implements, its shape, the constraints it respects: all of this is a direct consequence of the prompt that generated it.

A component without a corresponding prompt in `00_nucleo` is structurally illegitimate — not because it lacks documentation, but because its origin is untraceable. It is impossible to reproduce, audit, or consistently evolve what has no registered origin.

---

## Nucleation Lock

> **No prompt here → no code can be generated.**

Before any code generation in `01_core`, `02_shell`, `03_infra`, or `04_wiring`, a corresponding prompt must exist in this directory.

When an AI agent receives a task, its first action is to verify if a prompt exists in `00_nucleo/prompts/` for the target component:
- **Exists** → read the prompt, use it as context, generate or modify the code
- **Does not exist** → stop and request the developer to create the prompt first

---

## Structure

```
00_nucleo/
├── prompts/    # One file per component or feature
└── adr/        # Decisions affecting multiple prompts or the lattice
```

**`prompts/`** — Each file is the structured prompt that generated a component. It includes context, constraints, the instruction itself, the expected outcome, and the revision history.

**`adr/`** — Records architectural decisions that transcend a single component: lattice changes, adoption of new technology in L₃, redefinition of boundaries between strata, changes in global constraints.

---

## The Prompt as a Contract

A component's interface contract is not a separate document — it is the output of the prompt that generated it. The interface file in `01_core` generated by the prompt **is** the contract. `00_nucleo` holds the origin, not a prior description of the outcome.

This collapses three artifacts that traditional architectures keep separate — spec, contract, and code — into two: prompt and code. The relation between them is causal and direct.

---

## Evolution

When a component needs to change, the prompt changes with it. The revision is registered in the prompt file's history with date, reason, and affected files. The modified code remains traceable to its origin.

ADRs are created only when the change affects decisions that transcend the component — not for every code modification.

---

## Rules

1. **No executable code** — only `.md` prompts and ADRs.
2. **One prompt per component** — sufficient granularity for traceability, without excessive fragmentation.
3. **Human approval** — prompts require review before guiding generation.
4. **Mandatory history** — every prompt revision records date, reason, and affected files.
